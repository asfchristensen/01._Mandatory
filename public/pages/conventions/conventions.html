Conventions to follow



What are conventions?


A set of agreed-upon standards or guidelines that are followed by developers. The conventions cover various aspects fo software development e.g. coding style, naming conventions, file organization, documentation, testing and version control.


Code conventions


Having code conventions means that you have guidelines for a specific programming language that you must follow while coding. It gives the application structure, consistency and a better overview.

Examples of conventions in JavaScript:

- Variable/function names should be written in camelCase
- Constants should be written in CAPS, separated by "_"
- Use double quotes (") for strings
- Use strict equality (===) instead of soft (==)
- Use semicolons (;) after statements


Examples of conventions in Node.js:

- Use ECMAScript module format instead of CommonJS
- Use single quotes (') for strings
- Use four spaces for identation
- Always make a "app.js" file to hold the routes


Clean code


You basically write code for the "future you" and your co-workers. Not the machine. It states that all code should be readable and understandable. That's why you should never use abbrevations for variables e.g. footballTrainer should not be named ft etc. It's better to use clear and readable variable names rather than having a lot of comments in the code.


Clean code and code conventions is a good idea, because:


- It's good for documentation
- You don't need to know the code to know what it does
- It makes the code more: 

            - Readable
            - Maintainable
            - Reusable
            - Trustworthy

- It's easier to find bugs
- Others can read and understand the code and thereby decreasing the risk of errors
- Waste of time not having conventions (you spent time figuring out the code!)


Strict mode


The "use strict"; expression defines that the JavaScript code should be executed in a strict mode. It has a global scope and will only be recognized at the top of the script og function:
    
        "use strict";


        function printResult() {
            const x = 10;
            console.log(x); // 10
        }

        printResult();


        y = 20;
        console.log(y); // ReferenceError: y is not defined
This means that the code is now a stricter version of the language that provides additional safety checks and prevents certain actions that could lead to bugs or security vulnerabilities e.g.:

- Variables must be declared before they are used
- Cannot assign a value to an undeclared function
(prevents making global variabels by accident)
- Duplicate property names in objects are not allowed


A bad syntax is now being interpreted as errors, so the code should now be more "clean", realiable, predictable and secure.
